# Dev Notes: 2026-03-12

## Summary
This session moved the project from the legacy pairwise `fit_objects` workflow to a single integrated pipeline in `updated_workflow.Rmd` built around:

1. passage-level connected trees for karyotyped samples
2. simulation interval extraction from those trees
3. grouped Markov likelihood fitting for `p_mis` and `p_wgd`
4. forward simulation under the fitted parameters
5. Wasserstein-based permutation checks against observed endpoint karyotypes

The old `workflow.Rmd` / `R/analyse_lineage.R` path was removed so the repo now has one active analysis path instead of two partially overlapping ones.

## Tree and interval logic
- Fixed the connected-tree construction so each plotted tree is trimmed to the MRCA of the sampled passages rather than extending back to the connected-component root.
- Added component diagnostics to `updated_workflow.Rmd` so trimmed pre-MRCA ancestry is reported explicitly.
- Added media annotations to `db_col$samples` as `media_id`.
- Added `build_media_col()` in `R/db_utils.R` to subset the media table, rename `id -> media_id`, and drop non-varying columns.
- Added condition assignment to passage trees and updated the tree plotting so:
  - ordinary passages are colored by condition
  - karyotyped passages are plotted in red
  - legends were simplified to avoid misleading shape/size guides

## Simulation intervals
- Added `extract_karyotyped_intervals()` in `R/db_utils.R`.
- The interval logic now supports:
  - direct karyotyped ancestor -> descendant intervals
  - a one-step alias rule where moving the start up by exactly one passage makes it a valid start
- Each interval now carries:
  - `start`
  - `end`
  - `start_alias`
  - `n_collapsed_passages`
  - `conditions`
- Added `build_simulation_intervals()` to convert connected-tree intervals into simulation-ready records.
- Added `group_simulation_intervals()` to group intervals by `connected_tree` and resolved condition.
- Karyotypes are now loaded from the karyotype table into each interval, and are trimmed to chromosomes `1:22`.

## Markov fitting changes
- Replaced the old simulation-based likelihood as the main estimation path with a deterministic Markov likelihood over per-chromosome marginal copy-number distributions.
- Used the corrected per-copy missegregation logic from `R/karyotype_markov_vs_lineage.R`, fixing the old issue where higher-copy chromosomes were not given proportionally higher missegregation rates.
- Added support for fitting `p_wgd` jointly with `p_mis`.
- Group likelihood is now:
  - `LogL(group, p_mis, p_wgd) = sum(LogL(interval_i, p_mis, p_wgd))`
- Groups are defined by:
  - `connected_tree`
  - resolved interval condition

## Zero-copy state and diagnostics
- The Markov model now explicitly allows state `0`.
- This was done to match the observed data, but the consequence is important:
  - under the current `x * p_mis` rule, state `0` is absorbing
  - over long intervals or at high rates, the model can accumulate unrealistic mass on chromosome-loss states
- Added workflow diagnostics to make this visible:
  - `zero_copy_summary` listing which samples/chromosomes already contain zero-copy states
  - ploidy diagnostic plots comparing observed endpoint ploidy to predicted ploidy, including best predictions per `p_wgd` level

## Forward simulation changes
- Updated `ksim2.cpp` so the forward simulator now matches the fitted model more closely:
  - copy-number-dependent missegregation probability
  - zero-copy chromosomes remain absorbing
  - added a `p_wgd` path
- Added forward-simulation wrappers in `R/estimate_pmis.R`:
  - `simulate_interval_forward()`
  - `simulate_group_forward()`
- Added a demonstration chunk in `updated_workflow.Rmd` that:
  - selects one fitted group
  - runs forward simulations for that group
  - reports summary statistics
  - plots simulated vs observed ploidy

## Validation changes
- Added `wasserstein_permutation_test()` in `R/estimate_pmis.R`.
- This compares observed and simulated endpoint whole-karyotype distributions using a permutation test on Wasserstein distance.
- The helper now normalizes masses to probability weights before computing Wasserstein distance, so observed and simulated sample sizes do not need to match.
- The forward-simulation chunk reports:
  - Wasserstein statistic
  - permutation p-value
  - a boolean pass/fail flag
  - a compact interval-level summary

## Why these changes were made
- The old repository had two competing interval/fitting workflows, which made it unclear which outputs were current.
- The old likelihood was based on stochastic forward simulation and a single `p_mis`, which was slower and less aligned with the desired grouped inference.
- The new grouped Markov likelihood is faster, clearer, and is a better foundation for future hierarchical modeling.
- Adding `p_wgd` was necessary because the fitted ploidy and endpoint behavior indicated that missegregation alone was not sufficient.
- Updating the forward simulator ensures that validation uses the same assumptions as the fitted model.

## Outstanding issues / next steps
- The zero-copy absorbing state is still biologically troubling.
  - It is currently intentional and consistent across the Markov and forward models.
  - A future revision may need a different transition model if recovery from zero-copy should be allowed.
- The current `p_wgd` transition follows the user-specified matrix exactly, including the fact that only half of the WGD mass lands on `2i`.
  - If this was not the intended biology, `G` should be revisited explicitly.
- The Wasserstein permutation test is expensive.
  - It currently dominates runtime more than the forward simulation itself in many cases.
  - Likely optimization target: cache support coordinates and permute only masses.
- The forward simulation chunk is still a demonstration chunk for one group.
  - A generalized validation/reporting pass across all groups has not yet been built.
- The Markov fitting and forward-validation code now coexist in `R/estimate_pmis.R`.
  - If the workflow stabilizes, it may be worth splitting this into separate files for fitting vs forward validation.
